# Quickshare contract

The one document that pins how the pieces talk. The backend, the frontend
networking layer, and any UI (including one generated by Claude Design) all
depend on this and nothing else.

|--------|---------------------|---------|

## Signaling events (browser ‚Üî Flask, Socket.IO)

A dumb relay. It tracks room membership and forwards opaque WebRTC payloads.
Firebase mode mirrors these exact events client-side via Firestore.

**client ‚Üí server**
- `join`   `{ room, name }`
- `signal` `{ to, data }`  ‚Äî relay `data` to peer whose id == `to`
- `leave`  `{}`

**server ‚Üí client**
- `welcome`     `{ id, peers: [{ id, name }] }` ‚Äî your id + who's already here
- `peer-joined` `{ id, name }`
- `peer-left`   `{ id }`
- `signal`      `{ from, data }`

Convention: the **newer** peer always initiates the WebRTC offer.

## UI contract ‚Äî `useQuickshare()`

The UI imports `useQuickshare()` and renders from its return value. It must not
touch the socket, Firestore, or RTCPeerConnection directly.

```ts
{
  me: { id: string, name: string, color: string } | null,
  connected: boolean,
  signalingKind: "socketio" | "firebase" | null,
  roomId: string | null,
  roomUrl: string | null,            // share this to pair

  peers: Array<{
    id: string,
    name: string,
    color: string,                   // stable hsl() per peer
    status: "connecting" | "ready" | "failed",
  }>,

  transfers: Array<{
    id: string,
    peerId: string,
    peerName: string,
    direction: "send" | "receive",
    name: string,                    // file name
    size: number,                    // bytes
    mime: string,
    progress: number,                // 0..1
    status: "offered" | "transferring" | "complete" | "declined" | "failed",
    url?: string,                    // present on completed RECEIVES ‚Üí download
  }>,

  // actions
  sendFiles(peerId: string, files: FileList | File[]): void,
  acceptTransfer(transferId: string): void,   // receiver accepts an "offered"
  declineTransfer(transferId: string): void,
  saveTransfer(transferId: string): void,     // download a completed receive
  clearTransfer(transferId: string): void,    // dismiss from the list
}
```

### State rules the UI should honor
- A peer is only a valid send target when `status === "ready"`.
- A `receive` transfer starts as `offered` ‚Üí show **accept / decline**.
- After accept it goes `transferring` (watch `progress`) ‚Üí `complete`.
- A completed `receive` exposes `url` ‚Üí show **save**.
- A `send` transfer is `offered` until the peer accepts, then `transferring` ‚Üí `complete`, or `declined`.
